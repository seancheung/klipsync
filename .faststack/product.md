# KlipSync · 产品设计文档

> 上游：[需求](./requirements.md) ｜ 下游：[UI 设计](./design.md)

## 1. 产品概述

KlipSync 是一个部署在家庭 NAS 上的 **Web 优先共享剪贴板**。本文档把 [需求文档](./requirements.md) 中的 FR-xxx 逐条落地为可绘制的页面与可实现的交互。

**核心数据关系**：`用户(User) → 空间(Space) → 多个剪贴板(Clipboard) → 0~N 附件(Attachment)`。剪贴板即最小单位，无"历史条目"概念。

**设计总原则**：

- **Web 优先 · 响应式**：同一份前端代码覆盖桌面浏览器与移动浏览器，桌面走两栏（列表 + 编辑），移动走栈式（列表 ↔ 编辑）。
- **零显式保存**：编辑区内容自动落盘；空内容不生成剪贴板（FR-014 / FR-015 的硬约束贯穿整个编辑流程）。
- **"激活"是一等概念**：任意时刻主工作台只处于三种状态之一 —— `空白编辑区` / `激活已存在的剪贴板 X` / `列表返回态（仅移动端）`。多端同步、二维码、删除提示都围绕"当前激活对象"展开。
- **权限最小化**：普通用户只能操作自己空间；管理员只有账号级管理权，**看不到任何用户的剪贴板内容**（FR-053 硬约束）。

## 2. 信息架构

```
/
├── /setup                   首次部署引导（仅系统尚无 Admin 时可达）
├── /login                   登录
├── /force-reset             强制改密（FR-005，登录后若账号带 must_reset 标志则强制跳此）
│
├── /                        主工作台（默认进入"空白编辑区"或"最近激活剪贴板"）
│   ├── /c/new               显式新建 → 空白编辑区
│   └── /c/:clipboardId      激活某剪贴板
│
├── /settings                个人设置（修改密码 / 登出）
│
├── /admin/users             用户管理（仅 Admin）
├── /admin/system            系统设置（仅 Admin）
│
└── 通用状态页
    ├── 剪贴板不存在或无权访问（FR-040 扫码越权、URL 输错、对象已被另一端删除时的软落地）
    └── 未登录（跳 /login 并回带 next 参数）
```

模块树（组件维度，非路由）：

```
AppShell
├── TopBar（用户菜单 / 管理员入口 / 登出）
├── ClipboardList         ← P-004 主工作台左栏（桌面）/ 独立视图（移动）
│   ├── ListItem（文本摘要 · 附件计数 · 更新时间 · 置顶旗标）
│   └── EmptyStateHint
└── ClipboardEditor       ← P-004 主工作台右栏（桌面）/ 独立视图（移动）
    ├── TextArea（自动保存）
    ├── AttachmentGrid（缩略图 · 下载 · 单个删除）
    ├── Toolbar（复制文本 · 置顶 · 清空 · 删除）
    ├── QRCodePanel（二维码 + 当前链接，FR-040）
    └── DeletedInOtherTabBanner（FR-032）
```

## 4. 功能规格

### 4.1 首次部署引导（FR-062）

**目标**：确保首次启动 Docker Compose 后，任意设备访问都被强制落到"创建管理员"流程，避免空系统被任何人接管。

**入口**：系统启动后检测到数据库中 `users` 表为空 → 任何 URL 访问都 302 到 `P-001`。

**主流程**：

1. 用户访问任意路径 → 系统判定无 Admin → 重定向到 `/setup`。
2. P-001 展示表单：用户名（3–32 字符）、密码（≥ 8 位）、确认密码。
3. 提交 → 创建 Admin 账号（无 `must_reset` 标志）→ 自动登录 → 进入 `P-004` 空白编辑区（FR-012）。

**异常流程**：

- 两次密码不一致 → 提交按钮禁用 + 表单内行级错误。
- 用户名格式非法 / 过短 → 行级错误。
- 创建过程后端失败（磁盘写入异常等） → Toast 错误提示，不跳转，保留表单已填内容。
- 系统已存在 Admin 的情况下直接访问 `/setup` → 302 到 `/login`，不暴露 `/setup` 的存在。

**空 / 加载 / 错误状态**：

- 加载：提交按钮转圈 + 禁用，防重复提交。
- 错误：Toast + 行级错误两路并存，Toast 负责网络 / 服务器错误，行级错误负责字段校验。

### 4.2 登录（FR-001, FR-002, FR-004）

**目标**：把合法凭据换成一个长期有效的会话，送用户进入自己的空间。

**入口**：未登录者访问任意需要登录的路由 → 302 到 `/login?next={原路径}`。

**主流程**：

1. 用户在 P-002 填写用户名 / 密码，勾选"记住我"（默认勾选）。
2. 提交 → 后端校验 → 成功则下发长期 Cookie / refresh token（FR-004）。
3. 若账号 `must_reset = true` → 302 到 `/force-reset`（FR-005）。
4. 否则 302 到 `next` 或默认 `/`（主工作台）。

**异常流程**：

- 凭据错误 → Toast "用户名或密码错误"（不区分两者，减少枚举攻击面）。
- 连续错误达阈值（由技术方案定，产品层不关心具体数字） → 后端可临时锁，UI 表现为同样的 Toast + 按钮短暂禁用并倒计时。
- 扫二维码进入但未登录 → 登录成功后仍回落到该 `/c/:clipboardId`；若该剪贴板不属于当前账号，跳 P-008（FR-040 的越权分支）。

**空 / 加载 / 错误状态**：

- 首次打开页面：不做占位，直接渲染表单。
- 提交中：按钮转圈、表单整体禁用。
- "记住我"取消时：仍登录，但仅发短期 Cookie。

### 4.3 强制改密（FR-005）

**目标**：阻断"初始密码 / 管理员重置密码"在系统中长期存在的风险面。

**入口**：登录成功后若账号带 `must_reset = true`，任何路由访问都被 302 到 `/force-reset`；页面上除"登出"外无任何跳出入口。

**主流程**：

1. 用户在 P-003 填写旧密码（即初始密码 / 重置后密码）、新密码、确认新密码。
2. 提交 → 后端校验旧密码 + 更新新密码 + 清除 `must_reset` 标志。
3. 302 到 `/` 主工作台。

**异常流程**：

- 旧密码错误 → 行级错误，不跳转。
- 新密码等于旧密码 → 行级错误"新密码不能与旧密码相同"。
- 新密码 / 确认不一致 → 行级错误。

**空 / 加载 / 错误状态**：

- 顶部显示只读提示："你的密码为初始密码 / 已被管理员重置，请立即修改后继续使用。"
- 用户若点登出 → 销毁会话回 `/login`（允许退出，但不允许绕过改密进入系统）。

### 4.4 主工作台 — 剪贴板列表（FR-010, FR-011, FR-012, FR-031）

**目标**：让用户在三屏内看到"自己空间的全部剪贴板"，并以最短路径激活任意一条或新建。

**入口**：P-004 默认左栏（桌面）/ 默认首屏（移动）。

**列表项渲染**：

- 文本摘要：正文前 2 行（CSS 折行截断），若全为附件则显示占位"（仅附件）"。
- 附件计数图标：`📎 N`，`N = 0` 时不渲染图标。
- 最后更新时间：相对时间（"刚刚 / 3 分钟前 / 今天 14:22 / 2026-04-20"）。
- 置顶旗标：左上角小图钉。

**排序规则**：置顶（按置顶时间倒序） → 其余（按最近更新时间倒序）。

**主流程**：

- 点击列表项 → 路由切到 `/c/:id`，右栏（桌面）/ 整屏（移动）切到 P-004 编辑视图（FR-011）。
- 点击列表顶部"＋ 新建" → 路由切到 `/c/new` → 右栏呈现空白编辑区（FR-012）。
- 空间为空 → 列表区仅渲染"＋ 新建"按钮和引导文案"还没有剪贴板，开始输入即可创建第一个"，同时右栏默认就是空白编辑区。

**多端同步（FR-031）**：

- 另一端创建 / 删除 / 置顶变更 → 当前端列表在秒级内重排（流畅动画，不是整列表闪动）。
- **列表不下发"内容被更新"红点**（需求明确放弃该信号）。

**空 / 加载 / 错误状态**：

- 初次拉取列表：骨架屏（3~5 条灰块）。
- 拉取失败：顶部红色横幅 + "重试"按钮；既有渲染不清除。
- 长列表（>100 条）：不做虚拟列表首期，按需求规模可接受（单用户自用场景）。

### 4.5 主工作台 — 打开已存在剪贴板（FR-011, FR-013, FR-016, FR-030, FR-032, FR-040）

**目标**：进入后就是可编辑、可复制、可同步、可删除的激活状态，零操作负担。

**入口**：点击列表项 / 扫码 / 直接输入 URL `/c/:id` / 从另一端被同步激活。

**主流程**：

1. 路由切到 `/c/:id` → 拉取剪贴板全量（文本 + 附件元信息列表）。
2. 文本区渲染为可编辑 `<textarea>` 等价物，附件区渲染为缩略图网格。
3. 工具栏可见操作：**一键复制文本**（FR-016）、**置顶 / 取消置顶**（FR-021）、**清空**（FR-022）、**删除整条**（FR-018）。
4. 二维码面板展示当前 URL 的二维码（FR-040）。
5. 用户编辑文本 → 防抖自动保存（文本流 FR-013）。另一端秒级收到更新（FR-030）。

**异常流程**：

- 剪贴板 ID 不属于当前账号 → 跳 P-008。
- 剪贴板在另一端被删除（FR-032） → 编辑区顶部贴出横幅："此剪贴板已被删除（由其他设备）"，两个按钮：`返回列表` / `把当前草稿作为新剪贴板保存`。横幅出现后，本地文本区被置为只读（避免继续改一个已不存在的对象），附件缩略图灰化。
- 保存失败（网络抖动） → 文本区右下角出现"未保存 · 重试中…"指示；连续失败 > 10s 后升级为红色提示 + 手动重试按钮。
- 复制文本到系统剪贴板失败（浏览器权限被拒） → Toast "复制失败，请手动选中文本复制"。

**空 / 加载 / 错误状态**：

- 加载中：文本区骨架 + 附件区占位。
- 文本为空、附件也为空（刚清空完但还没离开）：编辑区正常可用，占位符"开始输入…"。
- 加载失败（剪贴板在服务端找不到了）：直接跳 P-008，不保留在本路由。

### 4.6 主工作台 — 空白编辑区与自动落盘（FR-012, FR-014, FR-015）

**目标**：消除"保存"这一心智，让"打开就开始输入"是唯一路径。

**入口**：路由 `/c/new`，或首次登录 / 空间为空时默认进入。

**主流程**：

1. 进入 `/c/new` → 右栏（桌面）/ 整屏（移动）渲染空白 TextArea + 空附件区。
2. 用户输入任意文本 **或** 上传任意附件 → 触发 **首次落盘**（FR-014）：
   - 后端创建一条新剪贴板，返回新 ID。
   - 前端把路由悄悄替换为 `/c/:newId`（`history.replaceState`，不产生新历史条目）。
   - 从此点开始，该视图等价于 §4.5 的"已激活剪贴板"状态，后续编辑走防抖自动保存。
3. 列表秒级出现这条新剪贴板（FR-031）。

**异常流程**：

- 用户在 `/c/new` 没输入任何文本也没上传任何附件就离开（关标签、切到别的剪贴板、切到设置页） → **不创建**剪贴板（FR-015）。
- 首次落盘请求失败：
  - 文本输入：保留用户输入在本地，顶部显示红色"保存失败，重试中…"，网络恢复后自动补建。
  - 附件上传：保留附件的 upload 任务条，网络恢复后自动续传；若附件超过配置上限（FR-054），前置拒绝并提示"单文件超过 XXX MB 上限"，不触发落盘。
- 用户在落盘还没返回 ID 就飞快开始编辑第二段 → 前端本地持续累计变更，等 ID 返回后整体提交（不生成两条剪贴板）。

**空 / 加载 / 错误状态**：

- 纯空白：TextArea 占位符"开始输入…"，附件区提示"粘贴、拖放或点击选择附件（单文件 ≤ XXX MB）"。
- 首次落盘中：状态指示"创建中…"，不阻塞用户继续输入。
- 失败重试中：不阻塞输入，错误横幅可关闭。

### 4.7 附件操作（FR-017, FR-019, FR-020）

**目标**：让附件与文本在同一"剪贴板"单位下被一并管理，不引入独立的"文件箱"。

**入口**：P-004 激活的剪贴板编辑区附件区。

**主流程**：

- **上传（FR-017）**：
  - 桌面：在编辑区内 `Ctrl+V` 粘贴截图 / 剪贴板文件；拖放文件到附件区；点"添加附件"按钮走系统选择器。
  - 移动：点"添加附件"按钮 → 弹出系统选择器（相册 / 文件）。
  - 上传中显示进度条；失败可点"重试"。
  - 允许任意文件类型；超过 FR-054 配置上限前置拒绝。
- **下载（FR-019）**：点附件卡片的"下载"按钮 → 浏览器标准下载流程。
- **删除单个附件（FR-020）**：附件卡片悬浮（桌面）或长按（移动） → 出现删除按钮 → 二次确认后仅移除该附件，文本与其他附件不受影响。

**异常流程**：

- 同名文件二次上传 → 允许并存（文件以内部 ID 区分），UI 上保留各自原文件名。
- 移动端相册选择多文件 → 队列化上传，逐一显示进度。
- 单文件超上限 → 在选择/拖放阶段就拒绝，不消耗上传配额。

**空 / 加载 / 错误状态**：

- 附件区为空：占位提示，不显示 0 条的空网格。
- 上传中：占位缩略图 + 进度 + 取消按钮。
- 上传失败：占位缩略图变红，"重试" / "取消"两按钮。

### 4.8 剪贴板级操作（FR-018, FR-021, FR-022）

**目标**：覆盖"置顶 / 清空 / 删除"这三种典型整理动作，避免用户在多操作中混淆意图。

**入口**：P-004 编辑区工具栏。

**置顶 / 取消置顶（FR-021）**：

- 工具栏图钉按钮 → 即时切换，无二次确认。列表端秒级反映（FR-031）。

**清空（FR-022）**：

- 工具栏"清空"按钮 → **二次确认** "确认清空？文本和所有附件将被删除，剪贴板本身保留。"
- 确认后文本置空、附件全删；剪贴板 ID 不变，仍停留在 `/c/:id`。

**删除整条（FR-018）**：

- 工具栏"删除"按钮 → **二次确认** "确认删除此剪贴板？该操作不可撤销。"
- 确认后剪贴板及附件被删除；当前路由切回 `/`（空白编辑区或最近的其他剪贴板，见下）。
- 当前端的"下一步落点"策略：若列表仍有其他剪贴板 → 激活排序最靠前的一条；否则落到空白编辑区。
- 多端一致性（FR-032）：另一端若正打开该剪贴板，走 §4.5 的"已被删除"横幅。

**异常流程**：

- 删除请求失败：Toast 报错，不离开当前路由；重试。
- 另一端先于本端删除了同一条：本端点删除时后端返回 404 → 视为"已经被删除"，走 FR-032 横幅提示 + 切回列表，而不是报错。

### 4.9 二维码快速跳转（FR-040）

**目标**：让用户把桌面当前剪贴板"一瞬间"带到手上的任意设备，不依赖任何客户端。

**入口**：P-004 编辑区右上 / 工具栏旁 `QRCodePanel`。

**主流程**：

1. 用户激活任意剪贴板 → 二维码面板自动生成"当前 URL（`/c/:id`）"的二维码。二维码编码的 URL 基址优先使用管理员在 P-007 配置的"公开访问基址"（FR-055），未设置时自动使用当前请求的访问地址作为基址。
2. 用户用手机系统相机扫码 → 手机浏览器打开同一 URL。
3. 手机未登录 → 落 `/login?next=/c/:id` → 登录成功后进入该剪贴板。
4. 手机已登录 **同一账号** → 直接进入该剪贴板。
5. 手机登录的是 **另一账号**（比如家人共用了手机浏览器） → P-008 "此剪贴板不存在或无权访问"。

**异常流程**：

- 二维码面板在空白编辑区（`/c/new` 还没落盘）不显示或显示为灰态（"开始输入以生成链接"），避免出现一个立即失效的二维码。
- 剪贴板 ID 刚在另一端被删除 → 扫码端打开后走 §4.5 的"已被删除"分支或 P-008（取决于扫码者是否为所有者）。

**空 / 加载 / 错误状态**：

- 二维码生成中（首次渲染）：占位方格。
- 点击二维码可弹出放大预览（方便远距离扫码）。
- 移动端默认折叠二维码面板（手机扫手机的场景极少），桌面端默认展开。

### 4.10 个人设置（FR-002, FR-003）

**目标**：让用户独立地改自己的密码和下线，不把这些混在主工作台的工具栏里。

**入口**：TopBar 用户菜单 → "设置"。

**主流程**：

- **修改密码**：旧密码 / 新密码 / 确认新密码 → 提交 → Toast "密码已更新"。
- **登出**（FR-002）：点"登出" → 二次确认 → 清会话 → 跳 `/login`。

**异常流程**：

- 旧密码错 / 新密码过短 / 两次不一致 → 行级错误。
- 登出请求失败（网络） → 前端直接清本地会话 + 跳 `/login`（宁可断联，也不卡用户）。

### 4.11 用户管理（Admin · FR-050, FR-051, FR-052, FR-053）

**目标**：让管理员在一个页面里管完家庭成员的所有生命周期动作，且无法触及任何剪贴板内容。

**入口**：TopBar 管理员菜单 → "用户管理"。非 Admin 访问 `/admin/users` → 直接 403 回 `/`。

**页面结构**：顶部"＋ 新增用户"按钮 + 用户表格（用户名、剪贴板数、存储占用、操作列）。

**主流程**：

- **新增用户（FR-050）**：点"＋ 新增" → 对话框填用户名 + 初始密码 + 确认密码 → 提交。新建用户默认 `must_reset = true`（强制首次改密，FR-005 挂钩）。
- **重置密码（FR-052）**：表格操作列 → "重置密码" → 对话框输入新初始密码 + 确认 → 二次确认"确认将该用户密码重置？重置后该用户下次登录将被强制改密。" → 提交 → 目标账号 `must_reset = true`。
- **删除用户（FR-051）**：表格操作列 → "删除" → **二次强确认**（要求再输一次用户名匹配）"确认删除用户 X？其空间的全部剪贴板与附件将被永久删除，不可撤销。" → 提交。
- **查看使用统计（FR-053）**：表格固定列展示"剪贴板数 / 存储占用"；**UI 上不存在任何"查看内容"的入口**。

**异常流程**：

- 用户名重复 → 新增对话框行级错误。
- 删除失败（数据库锁等） → Toast 报错，表格保留原行不消失。
- Admin 尝试删除 / 重置自己 → UI 禁用对应操作（防误删唯一 Admin）。

### 4.12 系统设置（Admin · FR-054, FR-055）

**目标**：让管理员在不碰 YAML 的前提下能调整附件上限，并显式设置用于生成二维码的公开访问基址。

**入口**：TopBar 管理员菜单 → "系统设置"。

**主流程**：

页面展示两个独立的可编辑卡片，各自独立保存：

- **附件大小上限**（FR-054）：数字输入（单位 MB，默认 10），"保存"按钮。合法范围（由技术方案敲定上下界，产品层要求 ≥ 1 MB）→ 保存 → Toast "已更新，新上限对**此后**上传生效，历史附件不受影响"。
- **公开访问基址**（FR-055）：文本输入（URL，默认空），说明文案"用于生成二维码中的链接；留空时自动使用当前请求的访问地址作为基址"。合法值必须以 `http://` 或 `https://` 开头，不含末尾 `/`，长度 ≤ 512 字符。保存 → Toast "已更新，对**此后**新生成的二维码生效"。

**异常流程**：

- 附件上限输入非数字 / 小于 1 → 行级错误。
- 公开访问基址格式非法（缺协议 / 带末尾斜杠 / 超长） → 行级错误。
- 保存失败 → Toast 报错 + 输入保留用户值。

### 4.13 无权访问状态页（P-008）

**目标**：为"对象不存在 / 不属于当前账号 / 已被删除"三种本质相似的失败路径提供统一软落地，避免让用户面对空白页或 HTTP 错误码。

**入口**：

- 扫码进入非己剪贴板（FR-040 越权分支）。
- URL 输错 / 剪贴板刚被另一端删掉（§4.5 / §4.8）。
- 路由到不存在的剪贴板。

**内容**：

- 主文案："此剪贴板不存在或无权访问"。
- 副文案：一行小字解释"可能是你扫到了别人账号下的二维码，或该剪贴板已被删除"。
- 主按钮："返回主工作台"。

## 5. 核心对象与状态

### 5.1 Clipboard 状态机

```
                ┌──────────────────┐
                │  Draft（空白编辑 │
                │  区，尚未落盘）   │
                └──────┬───────────┘
        文本/附件首输入 │
                       ▼
                ┌──────────────────┐
                │   Persisted      │
                │  （已存在服务端） │◀──清空 FR-022────┐
                └──┬──┬──┬─────────┘                 │
         用户删除 │  │  │ 另一端删除                   │
          FR-018 │  │  │  FR-032                     │
                 ▼  │  ▼                            │
          ┌──────────────────┐      清空动作会把文本/  │
          │     Deleted      │      附件清零但保留    │
          └──────────────────┘      Persisted 状态    │
                                                      │
                                    └─────────────────┘
```

- **Draft**：仅存在于浏览器内，没有 ID；离开页面即消失（FR-015）。
- **Persisted**：有 ID，多端同步的唯一生效状态（FR-030、FR-031 都只作用于 Persisted）。
- **Deleted**：服务端物理删除，前端若还持有该 ID 的打开态，走 FR-032 横幅。

### 5.2 User 状态机

```
[Created by Admin] ──must_reset=true──▶ [Login with initial pw]
                                                │
                                                ▼
                                 [Force Reset（P-003）必须完成]
                                                │
                                                ▼
                                         [Active · 正常使用]
                                                │
                                     Admin 重置密码 │
                                                ▼
                                  [回到 must_reset=true，下次登录重走]
```

- `Active` 与 `must_reset=true` 是挂在同一用户上的标志位，登录流程每次都检查。
- Admin 与普通 User 共用此状态机，Admin 账号由首次部署流程创建，首次创建不带 `must_reset`（由创建者自己定密码）。

### 5.3 激活态（前端视图态）

主工作台在任一时刻只可能是这三种激活态之一：

- `BlankDraft` — 路由 `/c/new`，未发生首次落盘。
- `ActiveClipboard(id)` — 路由 `/c/:id`，对应一条 Persisted 剪贴板。
- `ListOnly`（仅移动端） — 路由 `/`，尚未选择任何剪贴板（桌面端此态自动折叠为 `BlankDraft`）。

FR-030、FR-031、FR-032 的同步事件都要按当前激活态分发到正确的 UI 分支。
